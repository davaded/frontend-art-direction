# Visual Language

Use this reference when the UI is clean but visually generic, sterile, or lacking a memorable product world. Visual language is the system of imagery, material, color mood, texture, depth, iconography, illustration, photography, 3D, and graphic devices that makes a product recognizable.

This is separate from layout, typography, and component correctness. A product can have correct spacing and still feel visually empty.

## Visual Language Lock

For meaningful UI work, define this before polishing components:

```text
Visual metaphor:
Primary visual material:
Imagery/media direction:
Graphic device:
Depth and surface:
Color mood:
Texture/noise/pattern:
Icon/illustration/3D approach:
What to avoid:
```

The visual metaphor should come from the product's domain, content, data, or workflow. Avoid generic words like "modern" or "premium" unless they are backed by concrete visual choices.

## Reference Mix

Use 2-4 visual references, usually including:

- one product/UI reference for interaction and screen behavior
- one brand/identity reference for recognizability and graphic language
- one mood/material/media reference for image treatment, surface, depth, texture, or atmosphere

Borrow visual principles, not another brand's skin.

## Visual Language Moves

Choose a small number and apply them consistently:

- **Product object**: turn the core object into the visual anchor, such as a document, waveform, map, timeline, model, canvas, device, chart, or media object.
- **Graphic motif**: a recurring line, corner, grid, dot, mask, shape, crop, frame, glow, scanline, contour, waveform, coordinate, or symbol derived from the product.
- **Material system**: define how surfaces feel: paper, glass, brushed metal, soft plastic, dense terminal, editorial canvas, studio light, instrument panel, or cinematic screen.
- **Image treatment**: define crop, aspect ratio, contrast, color grade, depth of field, overlays, captions, borders, and background relationship.
- **Illustration or 3D language**: define whether objects are diagrammatic, cinematic, isometric, playful, technical, realistic, or abstract.
- **Depth logic**: decide what is flat, raised, inset, translucent, blurred, layered, or spatially separated.
- **Atmosphere**: define light, contrast, temperature, density, edge softness, noise, and background energy.
- **Data/AI/media visualization**: give invisible processes a specific visual form instead of generic pulsing dots or abstract blobs.

## Visual Coherence Checks

Before final delivery, inspect the screen and ask:

- Could a user recognize this product if the logo was removed?
- Does the visual language come from the product's content or workflow?
- Are images, icons, surfaces, and colors part of one system?
- Is there one memorable visual idea, or only clean components?
- Are decorative elements explaining, framing, or clarifying something?
- Does the visual style support trust and usability for this category?
- Does the design avoid generic stock imagery, generic gradient backgrounds, and random decorative shapes?
- Does dark or light mode keep the same product character?

If the answer is weak, remove generic decoration and build one stronger product-specific visual idea.

## Anti-Generic Rules

- Do not use abstract blobs, orbs, grid backgrounds, neon glows, or glass panels unless they express a product-specific idea.
- Do not use stock-like images when the user needs to inspect real product, data, people, place, media, or workflow.
- Do not use icons as decoration; use them to clarify action, category, state, or hierarchy.
- Do not mix illustration, 3D, photography, and abstract patterns without a system.
- Do not let a component library's default surface style become the visual identity.
- Do not add "premium" effects to compensate for weak composition or unclear content.

## Implementation Rule

If the UI feels visually lifeless:

1. Identify the product-specific metaphor or object.
2. Pick one visual material or graphic motif.
3. Apply it to the hero/primary workflow first.
4. Extend it lightly to navigation, cards, states, or empty screens.
5. Remove unrelated decorative effects.
6. Verify that the final screen still reads clearly and works at target viewport sizes.
